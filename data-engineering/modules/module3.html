Data Processing Layer

Mastering the fundamental models of modern data processing like processing models, processing guarantees, and problems.

The High-Level Processing Strategies

A. The Batch Model (Bounded)

Philosophy: "Data at Rest." You wait for all the data to arrive in a bucket (S3/HDFS), then you process the entire pile at once.

graph LR Start((Start)) --> Read[Read all data] Read --> Compute[Compute/Transform] Compute --> Write[Write Results] Write --> Stop((Stop))
  • The Model: Start → Read all → Compute → Write → Stop.
  • Key Strength: High throughput and 100% completeness.
  • Key Weakness: High latency (the data is always "old" by the time you see it).
🚀 Deep Dive: Beyond Map & Reduce

Modern engines like Spark, BigQuery, and Snowflake have evolved the legacy MapReduce pattern into DAG (Directed Acyclic Graph) execution.

  • The Legacy (MapReduce): A strict 2-step process that forced writing intermediate results to disk between phases, creating a massive I/O bottleneck.
  • Modern DAG Pipelining: Engines treat the entire job as one giant graph, keeping data in memory across multiple stages instead of forcing disk writes.
  • The Shuffle (The Real Bottleneck): This is the "Many-to-Many" network transfer. Data is re-partitioned by a key (e.g., user_id) across the network so that all records for that key land on the same worker for a Join or Aggregation. It remains the most expensive operation in any DAG.
graph TD S1[Stage 1: Scan] --> S2[Stage 2: Filter/Map] S2 -->|Shuffle: Network Transfer| S3[Stage 3: Reduce/Join] S3 --> S4[Stage 4: Finalize]

B. The Streaming Model (Unbounded)

Philosophy: "Data in Motion." You process each event as it arrives.

graph LR Event((Event)) --> Recv[Receive Event] Recv --> State[Update State] State --> Emit[Emit Result] Emit -.->|Continuous Loop| Recv
  • The Model: Continuous loop → Receive event → Update state → Emit result.
  • Key Strength: Minimum latency.
  • Key Weakness: Complexity in handling late data and out-of-order events.
🔄 Deep Dive: The Continuity Logic

In streaming, State and Checkpointing replace the "End of Job" signal found in batch processing.

  • State Management (The "Working Database"): The engine acts like a database, storing "Partial Aggregations" (e.g., sum of spend in the last 59 mins) in high-performance local stores like RocksDB.
  • Checkpointing (The Safety Net): Since streams never end, the engine periodically takes a Distributed Snapshot of its entire state and Kafka offsets.
  • Fault Tolerance: If the system crashes at 2:05 PM, it reloads the 2:00 PM checkpoint and replays the 5 minutes of data from the source to ensure no data is lost or double-counted.

The Modern Strategies

Beyond the simple "Batch vs. Stream" split, modern platforms use these specialized models:

C. The Micro-Batch Model

Philosophy: "Stream as a Sequence of Small Batches." Treats unbounded data as a continuous series of very small, discrete batch jobs.

graph TD Stream[Unbounded Stream] --> B1[Batch 1s] Stream --> B2[Batch 1s] Stream --> B3[Batch 1s] subgraph Processing Engine B1 --> P1[Process] B2 --> P2[Process] B3 --> P3[Process] end P1 --> Out1[Result 1] P2 --> Out2[Result 2] P3 --> Out3[Result 3]
  • The Model: Chop Stream → Run Mini-Batch → Commit → Repeat.
  • Key Strength: High reliability and deterministic fault tolerance (easy to re-run chunks).
  • Key Weakness: Higher latency than pure streaming (typically seconds vs. milliseconds).

D. The Unified Model

Philosophy: "Logic is Independent of the Source." Define the processing logic once and apply it to both bounded and unbounded data.

graph TD Source1[S3/HDFS - Bounded] --> API{Unified API} Source2[Kafka/PubSub - Unbounded] --> API API --> Engine[Processing Engine] Engine --> Result[Consistent Results]
  • The Model: Define "What, Where, When, How" → Map to Engine → Execute.
  • Key Strength: Operational simplicity; one codebase to maintain for all use cases.
  • Key Weakness: High implementation complexity and results depend heavily on the underlying Sink.

High-Level Processing Architectures (System Patterns)

These patterns define how you organize individual processing models into a complete, end-to-end system.

A. The Lambda Architecture

Philosophy: "Accuracy via Batch, Speed via Stream." Parallelize a "Batch Layer" for 100% correctness and a "Speed Layer" for real-time updates.

graph TD Source[Data Source] --> Batch[Batch Layer: S3/HDFS] Source --> Stream[Speed Layer: Kafka] Batch --> Master[Master Dataset] Master --> BatchView[Batch Views] Stream --> StreamView[Real-time Views] BatchView --> Serving[Serving Layer: Query Results] StreamView --> Serving
  • The Model: Batch (Immutable Master) + Stream (Delta Updates) → Merged View.
  • Key Strength: High fault tolerance; if the stream is buggy or crashes, the batch layer eventually corrects the data.
  • Key Weakness: Maintenance nightmare; you must maintain two identical codebases (e.g., Spark Batch and Flink Stream).

B. The Kappa Architecture

Philosophy: "Everything is a Stream." Treat historical data as just a long-running stream from the beginning of time.

graph LR Source[Unified Log: Kafka/Pulsar] --> Engine[Stream Processor] Engine --> Sink[Serving Layer/DB] Source -.->|Replay| Engine
  • The Model: Single Stream Source → Single Stream Engine → Final View.
  • Key Strength: Operational simplicity; only one codebase and one engine to maintain for all use cases.
  • Key Weakness: Requires high-retention stream storage (like Kafka Tiered Storage) to re-run historical data.

🧠 The "Hard Problems" in Distributed Processing

🕒 1. Watermarks & Late Data (The "Time" Problem)

The Challenge: In streaming, "Event Time" (when it happened) is rarely "Processing Time" (when the engine saw it). Data can arrive late due to network lag or mobile offline mode.

  • The Concept: A Watermark is a moving threshold that tells the engine: "I am reasonably sure no more data older than X will arrive."
  • The Solution: The engine uses the watermark to decide when to "close" a time window (e.g., 12:00-12:05) and emit the final aggregation.
graph LR E1(Event 12:01) --> P[Processing Engine] E2(Event 12:04) --> P W(Watermark 12:03) --> P P -->|Close Window| R[Emit 12:00-12:03 Results] E3(Late Event 12:02) -.->|Dropped/Side Output| P
⚖️ 2. Backpressure (The "Stability" Problem)

The Challenge: What happens when the Source (Kafka) sends 1M events/sec but your Sink (Database) can only write 100k events/sec?

  • The Concept: Without backpressure, the processing engine would buffer data until it ran out of memory (OOM) and crashed.
  • The Solution: The downstream component "signals" the upstream component to slow down, propagating all the way back to the source.
graph RL Sink[Slow Database] -- "Signal: I'm full!" --> Engine[Processing Engine] Engine -- "Signal: Slow down!" --> Source[Kafka Source]
🔐 3. Processing Semantics (The "Integrity" Problem)

The Challenge: Ensuring data is neither lost nor duplicated during a failure.

  • At-Least-Once: Simplest to build. Data is never lost, but might be processed twice. (OK for logs/dashboards).
  • Exactly-Once: The "Holy Grail." Requires a distributed transaction (Two-Phase Commit) between the engine and the sink. (Required for billing/finance).
graph TD A[At-Least-Once] -->|Retry| B[Possible Duplicates] E[Exactly-Once] -->|Checkpoint + Sink Commit| F[Single Logical Effect]
🔥 4. Data Skew (The "Hot Key" Problem)

The Challenge: In a distributed system, work is partitioned by a key (e.g., user_id). If one user (like a celebrity or a bot) has 1,000x more data than others, that single worker will bottleneck the entire pipeline.

  • The Concept: 99% of your workers finish in minutes, but the 1% handling the "Hot Key" takes hours, keeping the whole job "Running."
  • The Solution: Salting (manually spreading keys) or Adaptive Query Execution (AQE). Modern engines like Spark now dynamically detect skew at runtime and automatically split "Hot Partitions" into smaller chunks to balance the load across the cluster.

🏗️ Closing Thought

"The goal of modern processing is to make the distinction between 'Batch' and 'Stream' irrelevant to the business. In a mature platform, logic is defined once and executed against any dataset nature."

The Processing Manifesto

"By mastering stateful execution and unified APIs, you build pipelines that are as accurate as batch but as fast as streaming, providing the real-time heartbeat required for 2026 enterprises."
