Data Serving & Consumption

Designing the interface between the data platform and its diverse consumers—from Analysts to AI Agents.

Architectural Serving Patterns

Consumption is categorized by the "Serving Pattern." Each pattern maps a specific persona to a tailored technical configuration, ensuring the interface matches the latency and retrieval needs of the consumer.

1. The Analytical & BI Pattern

Consumer: Executive / Analyst

Architecture: Star Schema → Distributed SQL / OLAP. Focuses on high-speed join performance and human readability for decision support.

2. The Application & API Pattern

Consumer: The Application

Architecture: One Big Table (OBT) / Key-Value → API Gateway. Focuses on sub-millisecond response times and high concurrency for user-facing features.

3. The Intelligence & AI Pattern

Consumer: AI Agents / LLMs

Architecture: Embeddings / Feature Sets → Vector Search. Focuses on semantic retrieval (RAG) and training-serving consistency for ML models.

4. The Real-Time & Streaming Pattern

Consumer: The Application

Architecture: Live Streams → Stream Processing → Real-Time APIs. This pattern supports two delivery modes:

  • Push-based: Server initiates updates via WebSockets or SSE for instantaneous delivery (e.g., live fraud alerts).
  • Pull-based: Client fetches the latest state from a queryable stream table or materialized view (e.g., real-time dashboard polling).

The Three-Tier Serving Architecture

A mature serving layer decouples definition, structure, and execution to ensure architectural flexibility across all consumption patterns.

Tier 1: Semantic Layer (The Definition)

Metrics (e.g., Revenue, Churn) are defined once in code. This layer acts as the "translator" between technical data and business meaning, providing a consistent view for both humans and AI Agents.

Tier 2: Modeling Layer (The Structure)

Data is organized into formats based on retrieval: SQL Models (Star Schema, OBT), Vector Models (Embeddings), and Stream Models (Windowed aggregations and event envelopes).

Tier 3: Execution Layer (The Engine)

Engines are matched to the latency budget: Distributed SQL for exploration, Real-Time OLAP for dashboards, Specialized Stores (KV, Vector) for lookups, and Stream Processors for push/pull real-time serving.

🏗️ Closing Thought

"The success of a data platform is measured by its transparency. In a perfect serving layer, the Analyst sees a Star Schema, the App sees an API, and the User sees a Live Stream—none of them care about the underlying infrastructure."

The Serving Manifesto

"By providing a unified interface across request-response and streaming models, the platform remains the definitive 'Single Source of Truth' for the entire enterprise, regardless of how the data is being accessed or who is accessing it."
