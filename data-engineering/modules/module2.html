Module 2

Data Ingestion Layer

The Ingestion Layer's mission is to provide Reliability, Durability, and Decoupling. It is the "Buffer" that protects your analytical systems from the volatility of source systems.

High level Architecture (The Hub-and-Spoke)

In a modern stack, all ingestion strategies (CDC, Event-Driven, Batch, and API) ideally converge at the Event Streaming Backbone.

The Ingress

Data is captured from sources using the most appropriate strategy (Log-based for DBs, API-based for SaaS).

The Backbone

Data is held in a distributed, replayable log (Kafka/Redpanda). This handles backpressure and provides a "single source of truth" for raw data.

The Egress

Downstream systems (Data Lake, Real-time Analytics, Search Indices) consume the data at their own pace.

Ingestion Strategies

2. Event-Driven Systems (The PUSH Pattern)

Reactive architectures for low-latency event processing. In this strategy, the data platform is passive. The data is "born" in the application and sent immediately to an endpoint.

The Concept

The application code is instrumented to emit an event. This is common in Serverless or Microservices architectures.

Common Patterns

Webhooks: An external service (like Stripe or GitHub) calls your API when an event occurs.
SDKs/Trackers: Front-end applications (Web/Mobile) send clickstream data directly to an ingestion gateway (e.g., Segment, Snowplow).

Best For

Real-time user behavior tracking, third-party integrations, and decoupled architectures.

flowchart LR APP[Web/Mobile App] -->|SDK / Tracker| GW[API Gateway] EXT[External Service] -->|Webhook| GW GW -->|Event Push| KAFKA{Event Stream}

3. Log-Based Ingestion (Change Data Capture)

Instead of interacting with the database engine, you interact with the storage engine's history.

The Concept

Every modern database writes every change to a sequential file (WAL in Postgres, Binlog in MySQL, Redo Log in Oracle) before updating the table. CDC tools "tail" this file.

Strategy

You deploy a "Source Connector" (e.g., Debezium) that transforms these raw binary logs into structured events (JSON/Avro) and streams them to a broker.

Best For

Legacy modernization, synchronizing microservices, and high-volume transactional systems where you cannot risk performance degradation.

flowchart LR DB[(Operational DB)] -.-> WAL[WAL / Binlog] WAL -->|Tail Log| CDC[CDC Connector
e.g. Debezium] CDC -->|Stream Events| KAFKA{Event Stream}

4. Storage-Triggered Ingestion (The "Landing Zone")

This is a reactive pattern centered around Cloud Object Storage (S3, GCS, Azure Blob).

The Concept

A "Landing Zone" bucket acts as a buffer. As soon as a file is dropped (via SFTP, manual upload, or a legacy export), an event (like AWS S3 Event Notifications) triggers a serverless function (Lambda) to start the ingestion pipeline.

Strategy

Decouples the transfer of data from the ingestion of data. The source system doesn't need to know about your database; it just needs to know how to upload a file.

Best For

Partner data exchanges, legacy mainframe file drops, and massive bulk loads.

Landing Zone)] S3 -.->|Event Trigger| FUNC[Serverless Function] FUNC -->|Start Pipeline| PL[Data Pipeline]

5. API & Scraper Ingestion (The "Pull" Pattern)

This is the strategy for data you don't own.

The Concept

You have a scheduled orchestrator (Airflow, Dagster) that calls an external REST/GraphQL API or scrapes a website at specific intervals.

Strategy

Requires handling Rate Limiting, Pagination, and Authentication. You are at the mercy of the source's uptime and schema changes.

Best For

SaaS data (Salesforce, Zendesk), public datasets, and social media sentiment analysis.

e.g. Airflow] -->|Schedule Task| SCRIPT[Ingestion Script] SCRIPT -->|API Request| SAAS[External API
SaaS / Public] SAAS -.->|JSON Response| SCRIPT SCRIPT -->|Load| LAKE[(Data Lake)]

6. Gateway / Proxy Ingestion

Used for high-frequency, low-latency telemetry data (IoT).

The Concept

A specialized "Gateway" (like an MQTT Broker or an NGINX proxy) sits at the edge. It collects thousands of small packets from sensors, aggregates or validates them, and then batches them into the main data stream.

Best For

IoT devices, smart meters, and industrial sensor data.

e.g. MQTT Broker] GW -->|Batched / Validated| STREAM{Main Data Stream}

🚫 Ingestion Anti-Patterns

These are the "trap doors" that experienced engineers avoid when designing ingestion layers.

1. The "Dual-Write" Disaster

The Mistake: Trying to write to your Primary Database and your Message Queue/Stream in the same application code block.

# DON'T DO THIS
db.save(user_data)
kafka.send("user_topic", user_data)

The Risk: If the DB save succeeds but the Kafka send fails, your Data Platform is out of sync. The Fix: Use CDC to ensure the DB is the single source of truth.

2. The "Production Hammer" (Bulk Pulls)

The Mistake: Running heavy SQL queries against production replicas to extract data.

The Risk: Massive SELECT * scans can saturate I/O and cause replication lag. The Rule: If moving >10% of a table daily, move to CDC or incremental exports.

3. Ignoring "Schema Poison"

The Mistake: Allowing sources to send raw, unvalidated JSON into the stream.

The Risk: A sudden change (e.g., Integer to UUID) crashes downstream Spark/Flink jobs. The Fix: Use a Schema Registry (Avro/Protobuf) to enforce a contract at the Gateway.

4. The "Infinite Retry" Loop

The Mistake: Setting up an ingestion worker that retries a "poison pill" message forever.

The Risk: Pipeline gets stuck and lag grows indefinitely. The Fix: Implement Dead Letter Queues (DLQ) for manual inspection.

🏗️ Closing Thought

"Ingestion is the 'immune system' of your data platform. By favoring log-based capture over brittle batch pulls, you ensure that the downstream platform remains resilient to the volatility of source systems."

The Ingestion Manifesto

"The goal of a mature ingestion layer is to provide durability and decoupling. Every raw event captured today is a potential feature for an AI model tomorrow—capture it reliably, or lose the context forever."
